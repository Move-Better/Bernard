## 2026-07-22 — T3 slot model (fork found during grounding)
Grounding for T3 (`.claude/mockups/week-slots.html`) found that nothing in the data model persisted a weekday/time slot — `assignSlots()` computed a fresh day+hour every planning run with no persisted anchor, and `cadence_policy.channels[x].target_per_week` is a count, not a schedule. "Buffer-style slots" (the 7-21 decision) had two honest readings; resolved here.

- **Fixed slots, persisted.** `cadence_policy.channels[x].slots` is a real per-channel `[{weekday, hour, format, enabled}]` list, seeded from the existing `BEST_HOUR`/even-spread math so day-one targets translate into a sensible layout with no manual setup. Why: matches the literal Buffer mental model, AND gives T4's day/time learner a concrete place to write proposed changes.
- **Format label = Post / Reel / Story, not "Post/Carousel."** Verified against bundle.social's payload (`type: 'POST'|'REEL'|'STORY'`, no carousel type) and Meta's Graph API (`media_product_type`; carousel is a `media_type` detail of Feed, not a sibling category). Matches T2's `atom.format` enum exactly.
- **Mockup signed off 2026-07-22** (`.claude/mockups/week-slots.html`, 4 screens) — build below. Story stayed disabled per #2219 throughout; the slot system reads `cadence_policy.channels[x].enabled` generically, so Story gets working slots automatically whenever it's revived, no further calendar work needed.

## 2026-07-22 — Batch drafting: don't build it, widen the pre-draft window (challenge gate)
Brief + Q's answer (AskUserQuestion): **widen the window, no batch-draft UI.**

The 2026-07-22 UX pain check surfaced `Draft`-button dead clicks on `/week` and the first read was "users can't batch-draft the week." Grounding it killed that framing: batch pre-drafting **already exists** (`predraftWeek.js`, lane `pre_draft_week`, enabled on movebetter since 2026-07-03). The real defect was its eligibility window — `plan_week=eq.${nextMonday}`, a strict equality. The moment that week became the *current* week, every still-undrafted slot in it was permanently ineligible and could only be hand-drafted one at a time.

**Evidence:** 11 of movebetter's last 66 drafts (17%) carried the `voice_audit.predrafted` marker; the owner hand-clicked the other 83%. On 7-21, five `Draft` dead clicks each landed 12–39s *before* a real draft row (linkedin/gbp/instagram/instagram/linkedin) — one person drafting 5 posts serially over 11 minutes, hitting the single-flight lock on every attempt to start the next. The 7 atoms involved had existed since early June, so pre-draft had six weeks and still never became eligible for them.

**Change:** window is now `thisMonday <= plan_week <= nextMonday`, plus `scheduled_at >= now() - 24h` so long-past slots don't burn model calls (weeks before thisMonday stay excluded, so abandoned plans stay abandoned). Validated against prod before shipping: old filter matched **0** candidates, new filter matched **1** — the current week's remaining pending slot that would otherwise have waited for a hand-click.

**Per-tick cap deliberately NOT raised.** `agent-tick` runs `*/5`, so the existing cap of 2 drains a 10-slot week in ~25 min; the binding constraint is `daily_ai_call_cap` (40), not the cap. Raising it would add spend risk without throughput.

**Kill criterion:** if by **2026-08-19** (4 weeks) the agent-predrafted share hasn't moved from 17% to **≥60%** of drafts, the window wasn't the constraint — investigate the daily AI-call cap and the `interview_id`/`scheduled_at` eligibility filters before reconsidering any batch-draft UI.
