# Bernard AI Colorist — Concept Brief

_Generated 2026-06-19 by the ai-colorist-concept-research workflow (11 agents: product sweep → feasibility → synthesis). Read-only research. Spec basis for the colorist build._

## Concept
Bernard AI Colorist is a per-workspace "house look" engine that makes real clinical/movement photos look professionally graded — set once by describing a look in plain language or matching a reference photo, then applied deterministically and reproducibly to every photo the workspace bakes. The intent layer is conversational/example-driven (a multimodal model proposes a structured grade-parameter set from a description or a set of "this is our look" reference shots); the apply layer is pure Sharp math (exposure, contrast, saturation, white balance, tint, levels, local contrast, plus a 1D/3D LUT for the brand signature) that runs server-side in the existing compositor with $0 per-image cost and bit-for-bit reproducibility. The grade is non-destructive by construction: the original blob is never touched; a derived graded JPEG is written to a new blob path and the grade is stored as a small JSON object on the row, so any re-render is free and identical. The already-mocked structured slide/composition editor stays the manual backbone — text, headline, scrim, layout, and aspect remain human-driven; the colorist only governs the photographic grade underneath the overlay.

The owner's three hard requirements reconcile cleanly with feasibility: NON-DESTRUCTIVE is native (derived artifact + stored param object); BRAND-MATCH is the JSON "house look" preset persisted on the workspace row (the Imagen/Aftershoot "save-a-profile" pattern, but expressed as Sharp params, not a learned 5,000-image model Bernard doesn't have); and BACKGROUND/STYLE-ONLY is the one genuinely hard requirement. Honest framing: a global parametric grade (white balance, gentle exposure/contrast/saturation lift, a restrained LUT) does NOT meaningfully alter how a clinician looks when the parameters are clamped (saturation <=1.20, hue within +/-8 degrees, recomb near-identity) — it lifts the whole frame together. True "grade only the gym wall, leave the skin neutral" requires subject segmentation (a model) and is a later phase. The honest v1 promise is "professionally graded, brand-consistent, subject-safe by constraint" — not "background-isolated." Background-isolation and generative background replacement are real but deferred, behind clamps and explicit toggles.

## Architecture
End-to-end, three layers with a hard split between the model (proposes numbers) and Sharp (applies numbers deterministically):

1. INTENT (model proposes a grade param set). Two entry modes feed one output schema. (a) DESCRIBE: Q types "warm editorial, lift the shadows, slightly muted background" -> Claude (Opus 4.8 / Sonnet vision) returns a structured JSON grade object via the structured-output JSON-schema API (guarantees shape, no parse failures). (b) MATCH: Q marks 2-5 approved photos as "our look" -> the system computes per-channel Lab mean/stdev (Reinhard color-transfer statistics) deterministically from those references via sharp().toColourspace('lab').raw(), and/or hands the references to the model as cached system-prompt images (cache_control ephemeral) so it proposes params anchored to them. Either way the output is the same small JSON: { exposure, contrast, saturation, temperature, tint, highlights, shadows, levels, clahe, lut_ref }. Param ranges are tightly bounded in the schema so model drift is perceptually small and subject-safe clamps are enforced server-side at write time.

2. APPLY (deterministic Sharp pipeline). A new applyGradeParams(sharpInstance, params) helper replaces the single grade scalar at api/_lib/brandRender.js (~line 463-468, today: modulate({brightness:1+g*0.12, saturation:1+g*0.18})). Fixed operation order (matters — wrong order with a LUT produces garbage): primary correction (exposure via linear(2^ev,0); contrast via linear; white balance via per-channel linear or recomb 3x3; tint via tint/recomb) -> tone shaping (gamma; 1D per-channel curve via extractChannel+256-LUT buffer remap+joinChannel for highlights/shadows; clahe for local contrast) -> brand LUT (3D .cube via JS trilinear interp, ~40ms; the brand signature layer) -> THEN the existing SVG text/scrim overlay composite. Every param defaults to neutral so legacy rows (only grade set) re-render identically.

3. SAVE (per-workspace house look). The grade object persists in two places: the workspace-level default in a new workspaces.photo_grade_preset JSONB column (the "house look," resolved through the existing resolveBrandColors/workspace JSONB chain, cache-invalidated on save like other brand settings), and the per-photo override in the existing content_items.photo_treatment JSONB alongside headline/aspect/scrim. Non-destructive: original blob untouched; graded output written to media/graded/<workspace-id>/<asset-id>.jpg (or rendered on demand from stored params). Versioned as {v, params} so old bakes stay reproducible.

WHERE OPTIONAL BACKGROUND-GENERATIVE FITS (later, bolt-on, async): a v2/v3 "Clean/replace background" toggle inserts a segmentation step before the grade. Bria RMBG 2.0 (~$0.018/call, the only segmentation option with explicit commercial IP indemnification — essential for real clinical staff published under a practice brand) produces a subject mask cached per asset. Then EITHER (a) subject-safe differential grading: apply the full grade to background pixels only, composite the original subject back via Sharp mask composite (subject pixels untouched, deterministic), OR (b) Bria /replace_background (~$0.023/call, seed stored for reproducibility, mask-protected so subject is never mutated). This runs async via waitUntil with a mask-preview QA gate (edge fidelity on hair/equipment is the real failure mode), polled with the MediaDetail refetchInterval + hard-cap pattern. It is architecturally clean but NOT v1 — the deterministic grade ships first and stands alone.

## Competitor precedent
Borrow the SAVE-A-PROFILE + APPLY-DETERMINISTICALLY pattern from Imagen AI / Aftershoot (a brand "look" is a persisted profile applied consistently across a library) but ELIDE their learn-from-5,000-edits model — Bernard has no edited-photo back-catalog and doesn't need one; the look is a brand decision encoded as a small JSON param object, not a learned style. Borrow the TWO-PASS "normalize per-image, then stamp a fixed look" architecture from Imagen's Adaptive LUT and professional colorist workflow (primary correction to a neutral baseline FIRST, then the brand LUT) — this is what lets one fixed house look survive wildly varied phone exposures. Borrow the DESCRIBE-A-LOOK UX from VSCO Prompt and LUTgen.app, but resolve it to an inspectable numeric param set (not generative pixels) — this is the novel differentiator vs Capture One Match Look, which requires a visual reference; Bernard can accept plain text because the underlying schema is small and well-defined. Borrow the REFERENCE-TO-LUT path from Color.io / Colorby AI / Lumix Magic LUT for the "match our brand" entry mode (derive a .cube once from an ideal shot). Borrow SUBJECT/BACKGROUND DIFFERENTIAL GRADING from DaVinci Resolve Magic Mask + DxO PhotoLab 9 AI masking (separate grade stacks per region) and Luminar Neo Light Depth for the later background-isolation phase. AVOID as wrong-tool-for-the-job: Adobe Lightroom Firefly API (returns rendered JPEGs not params, per-call billing, breaks determinism); Photoshop Neural Filters and Flux Fill for color (generative, non-deterministic, and Flux has no commercial IP indemnity for real clinical people); Photoroom/Canva generative background replacement as a default (alters the clinical document). For any generative background work, Bria is the only IP-indemnified choice.

## Feasibility verdict

| Capability | Verdict | Note |
|---|---|---|
| Exposure (EV stops) | **cheap-deterministic-now** | sharp.linear(2^ev, 0) — exact, per-channel, no model |
| Brightness | **cheap-deterministic-now** | modulate({brightness}) — already wired to the current grade scalar; just promote to a first-class param |
| Linear contrast | **cheap-deterministic-now** | linear(a, 128*(1-a)/255) midpoint-preserving; true S-curve (toe+shoulder) needs the 1D-curve helper |
| Saturation | **cheap-deterministic-now** | modulate({saturation}) — HSL-space; keep range conservative (0.9-1.35) since HSL is perceptually uneven and over-saturates skin |
| White balance / temperature | **cheap-deterministic-now** | recomb 3x3 or per-channel linear; visually equivalent though not Kelvin-calibrated — label it 'warmth' in UI, not 'color temperature' |
| Tint (green/magenta) | **cheap-deterministic-now** | tint({r,g,b}) or recomb green column |
| Global hue rotation | **cheap-deterministic-now** | modulate({hue}) in degrees — clamp to +/-8 for subject safety |
| Levels (manual + auto) | **cheap-deterministic-now** | linear() for manual input/output range; normalise({lower,upper}) for auto-levels |
| Vignette | **cheap-deterministic-now** | SVG radial-gradient composite — already the compositor pattern |
| Highlights / shadows (zone) | **cheap-deterministic-now** | NOT native; needs a ~30-line 1D per-channel curve (extractChannel + 256-entry LUT buffer remap + joinChannel). gamma() is a poor proxy — build the curve helper, don't ship the proxy |
| Arbitrary tone curve (Bezier/spline) | **cheap-deterministic-now** | piecewise-linear via the same 1D-LUT helper; full cubic bezier not in Sharp but piecewise covers the need |
| 3D LUT / brand .cube look | **cheap-deterministic-now** | not in Sharp natively; ~50 lines of JS trilinear interpolation over a 33^3 grid (or image-lut/color-lut npm), ~40ms/image, $0, fully deterministic — the single highest-ROI step for 'match our brand look' |
| Clarity (mid-frequency local contrast) | **cheap-deterministic-now** | clahe() or large-sigma sharpen() — approximate but acceptable |
| Brand look derivation from reference photos (Reinhard color transfer) | **cheap-deterministic-now** | sharp.stats() + Lab mean/stdev computed at setup time (not render time); output is per-channel linear(a,b). Pure JS Lab pixel-walk is slow (~100-400ms) but runs once per look, not per render |
| Describe-a-look -> param set | **needs-model** | Claude vision + structured-output JSON schema. ~$0.08-0.15/photo proposal, or ~$0.01-0.05 one-shot at look-setup. Anchor with cached brand references; clamp ranges in schema |
| Auto-grade an incoming photo to neutral baseline | **needs-model** | either deterministic (sharp.stats histogram -> linear) for plain normalization, OR Claude vision returning {exposure,contrast,saturation,warmth} for scene-aware correction. The deterministic path is preferred for v1 |
| Vibrance (skin-protecting selective saturation) | **needs-model** | no clean Sharp path; per-pixel saturation gating. Approximations are bad enough to be worse than omitting — defer to a model phase |
| Selective hue shift (e.g. wall color only) | **needs-model** | per-pixel hue-range detection; needs a model or manual mask |
| Subject/background isolation (true background-only grade) | **needs-model** | segmentation required — Bria RMBG 2.0 (IP-indemnified, ~$0.018) or BiRefNet/SAM. Mask cached per asset; grade stays deterministic. This is the real answer to 'never alter the clinician' beyond clamps |
| Generative background replacement | **later** | Bria /replace_background (~$0.023, IP-indemnified, seed-stored). Async + mask-preview QA gate. Pin model version. NOT v1 |
| Dehaze | **later** | dark-channel-prior / dehazing CNN — full CV task, low priority for clinical content |
| Noise reduction | **later** | frequency-selective NR is a DNN task; sharp.blur() is not a substitute |
| Background relight (keep the room, fix the light) | **later** | no production API exists (research only — Lumen/Lux Post Facto); skip |

## Phasing
Phase 0 — Sharp grade engine, fully deterministic, $0/image (1-2 days, ~$3 Sonnet). Replace the single grade scalar in renderEditorialPhoto (and the two hardcoded whoopTemplates presets) with applyGradeParams(sharpInstance, params) covering exposure, contrast, saturation, warmth(WB), tint, hue, levels, clahe — each defaulting to neutral so legacy rows re-render identically. Add the ~30-line 1D-curve helper for highlights/shadows. Store the param object in content_items.photo_treatment. Verify locally with a node harness (per the WHOOP/Sharp CLAUDE.md rule — render every variant to /tmp and read the JPEGs); no browser needed. This alone is "AI colorist without the model" and covers ~65% of the Lightroom Basic panel deterministically. SHIP FIRST.

Phase 1 — Brand house look + LUT (1-2 days). Add workspaces.photo_grade_preset JSONB (migration with GRANT to service_role inline). Bake one "Bernard look" .cube from a reference Q approves; apply via JS trilinear interp as the final pre-overlay step. Wire the two-pass pipeline (normalise per-photo -> stamp the fixed look) so one preset survives varied phone exposures. Add the MATCH entry mode (Reinhard Lab stats from 3-5 approved photos -> preset). Enforce subject-safe clamps (saturation <=1.20, hue +/-8, recomb near-identity) server-side at write. Settings UI: upload references -> "Derive look" -> before/after preview. Mockup-first per CLAUDE.md before building the UI.

Phase 2 — Describe-a-look intent layer (2-3 days, ~$4-8). Add proposeGradeParams via Claude vision + structured-output JSON schema, references cached in the system prompt. "DESCRIBE" box ("warmer, more contrast, muted background") returns a param set the user reviews as pre-filled sliders before saving. Negligible per-call cost; the model proposes, Sharp applies. This is where it starts to feel like an AI colorist.

Phase 3 — Subject-safe background-only grading (3-5 days). Bria RMBG 2.0 segmentation at upload, mask cached per media_asset. Differential grade: full grade on background, original subject composited back untouched. Mask-preview QA gate before commit. This delivers the literal "never alter the clinician" promise.

Phase 4 — Generative background replacement (optional, demand-driven). Bria /replace_background behind an explicit toggle, async waitUntil + poll-with-hard-cap, seed + model-version pinned in photo_treatment, stored output as canonical. ~$4/mo at 100 photos. Only if owners ask.

Throughout: the structured slide/composition editor stays untouched as the manual text/layout backbone — the colorist governs only the photographic grade beneath the overlay.

## Risks

- BACKGROUND-ONLY is the headline promise but is NOT v1. Every Sharp op is global. Phases 0-2 lift the whole frame including the clinician's skin. Mitigate honestly: clamp params hard (saturation <=1.20, hue +/-8, recomb near-identity) so the subject is safe-by-constraint, and frame v1 as 'professionally graded, brand-consistent, subject-safe' — NOT 'background-isolated.' True isolation is Phase 3 (segmentation). Set this expectation with the owner before Phase 0 ships, or the demo will feel like a broken promise.
- Highlights/shadows via gamma() is perceptually wrong (flattens the whole image). Build the 1D-curve helper or omit the control — do not ship a crude proxy that trains users to expect bad output.
- HSL saturation (modulate) clips vivid blues/greens to neon before skin tones move — blue/green equipment and scrubs over-saturate first. Keep saturation range conservative; gate true vibrance behind a model phase.
- Model numeric output is not bit-reproducible (temperature -800 vs -750 across runs). Imperceptible at clamped ranges, but ALWAYS store and re-use the proposed params (never re-call the model on re-render) and never surface raw numbers to users.
- Batch drift: photos graded across a long session diverge unless every model call is anchored with the same cached brand references. Pin schema ranges tightly so drift is perceptually bounded; the deterministic per-workspace preset (Phase 1) sidesteps this entirely for the common path.
- Legacy compatibility: every existing content_item has only the grade scalar set. applyGradeParams must treat all new fields as optional-with-neutral-defaults and wrap (not replace) the current modulate line, or old posts re-render differently.
- Generative background IP exposure: real clinical staff published under a practice brand. Use ONLY Bria (licensed-data + commercial IP indemnification) for any segmentation or generation — never Flux Fill, Clipdrop, or DALL-E for this content.
- Segmentation mask edge quality is the real generative failure mode (resistance bands, foam rollers, hair in motion, skin-tone-matched wood backgrounds nick the foreground). A mask-preview QA gate before commit is non-negotiable for Phases 3-4; a mis-segmented shoulder publishes a missing-limb artifact.
- Vercel runtime: full seg + bg-gen pipeline is 10-20s — must run as a Node handler with waitUntil and a hard-capped refetchInterval poll, never on Edge (10s timeout) and never as a bare floating promise (frozen at response time).
- Per-workspace preset cache: photo_grade_preset save must call invalidateWorkspaceCacheBySlug or warm instances serve stale looks for up to 60s.
- LUT performance: 33^3 trilinear interp is ~40ms at 1080px but scales with resolution — benchmark before making it the default path and respect the existing 2000px long-edge cap.

## Open product questions

- Default ON or opt-in? Should the workspace house look auto-apply to every baked photo, or stay an explicit per-photo 'Apply look' action? (Auto = brand consistency by default but surprises owners; opt-in = safer rollout.)
- Is v1's 'subject-safe-by-clamp' global grade acceptable as the shipped promise, or is true background-isolation (Phase 3 segmentation) a hard gate before any colorist ships? This determines whether Phase 0 can stand alone.
- Primary intent mode for owners: DESCRIBE (type a look), MATCH (pick reference photos), or KNOBS (labeled sliders)? All share one schema, but which leads the UI shapes the first build.
- Who owns/locks the house look — workspace-admin-only, or can any user override per photo? And can an individual override drift from the locked brand look (governance vs flexibility)?
- One look per workspace, or named looks (e.g. 'clinical bright', 'warm editorial') the user picks per piece/campaign?
- Does 'match our brand look' mean match the brand color palette (emerald/Blue Spruce tokens already in brand.js) or match the photographic feel of approved reference shots? These are different derivations.
- Is the structured composition editor's preview WYSIWYG with the graded photo, or does the grade only appear at publish? (The carousel/preview-vs-published bug class in CLAUDE.md makes this load-bearing — the editor must render the SAME graded output that ships.)
- Appetite for the generative-background per-image cost (~$0.04 IP-indemnified) and the editorial-honesty question of replacing the real clinical setting at all — is that ever on-brand for clinical content, or should backgrounds only ever be graded, never replaced?
- Re-grade-on-edit behavior: when a photo's aspect or crop changes, re-derive params or freeze the stored graded artifact? (Seed-based generation is only 'similar,' not identical, on re-render.)
