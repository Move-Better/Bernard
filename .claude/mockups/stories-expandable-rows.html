Investigation · Stories table

Expandable per-channel rows on Stories

Can the Posts-tab expand pattern come to the main Stories table? Short answer: yes, and the data already fits — but one interaction decision decides whether it's safe. Below: the findings, then a live mockup of the recommended approach.

GO with caveats — don't hijack the row click
Findings — the five questions
1 · Row-click today

The whole row navigates to /stories/:id (StoryDetail), with a hover prefetch of that story and a useSmartBack return contract. This is heavily-used muscle memory. Mirroring Posts' "click = expand" here would delete that quick-nav.

Keep row→detail; add a separate expander
2 · Do pieces carry the fields?

story.pieces has platform / status / scheduled_at / published_at / performed_well — enough for status + time + an Edit button. Caption & media are stripped (view=card + summarizePiece). But the row already hover-prefetches fetchStory (full pieces incl. content) — reuse that cache for the preview.

Fields present; caption is opportunistic
3 · Volume & perf

movebetter: 16 stories, avg 5.9 / max 12 pieces each. Expand is opt-in per row and reuses the hover-prefetch cache, so no new N+1 beyond a fetch the row already fires. Trivial scale.

No perf concern
4 · Sort / filter / paginate

Key expand state by story.id in a Set (not row index). Re-sorts and filter changes then keep the right rows open and never jump. Page reset already fires on the result-set key.

Safe if keyed by id
5 · Duplicate of StoryDetail?

StoryDetail already shows every channel (AssetsPane) in depth. Expand-in-place is additive: a quick peek + a direct jump to one channel's editor without a full page nav. Real, but modest — worth shipping only as a lightweight lens, not a second editor.

Additive, keep it light
✓ Edit target already exists

/publish/:pieceId (StoryboardPublish) loads any single content_item and already handles both story pieces and post pieces. A sub-row "Edit" reuses the exact canonical editor — nothing new invented.

Zero new routes
Recommended interaction — try it

Click the › chevron to expand a story into its channels. Click anywhere else on the row to "open the story" (row-click stays intact).

The one caveat that makes it safe

On Posts, "click row = expand" was fine because the row only ever opened an editor. On Stories the row opens StoryDetail — a richer, more-used destination. So the recommendation diverges from the approved Posts interaction in exactly one way:

  • Add a dedicated chevron disclosure (leading cell). Chevron = expand; the rest of the row keeps its /stories/:id navigation, hover-prefetch, and back contract.
  • Lazy per-row: fetch full pieces only on first expand, from the same queryKeys.stories.detail cache the hover already warms.
  • Expand state = Set<storyId>, so sort/filter/pagination never lose or misplace an open row.
  • Mirror the approved Posts decisions otherwise: expand-only (no auto-open primary channel), no publish-all button.
Blast radius — what it touches
Files

One component: StoriesTableView.jsx. Optionally a tiny shared sub-row. No API, no migration, no route.

Data

Reuses existing fetchStory / useStory. No new query, no select-clause change.

Regression to watch

The chevron cell must stopPropagation so expanding never also fires row-navigation. That's the whole footgun.
